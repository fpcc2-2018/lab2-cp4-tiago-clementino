Abstract

A exploratory report involving an analysis task proposed by Wikimedias Foundation. The work described here aims to find specific relationships between the properties present in the data. The result is some useful conclusions that can be reached only on the basis of the available information.

Objetivo

Vamos reproduzir um problema de explorao de dados que a Fundao Wikimedia usou em 2016 no recrutamento de pessoas para trabalhar em seu departamento de descoberta. O objetivo seguir as instrues colocadas na tarefa original, respondendo as 4 perguntas que so colocadas.

A entrega consiste de uma verso de um repositrio no Github com um link para o seu relatrio final no incio do README.md do seu repositrio.

Contexto

O departamento de descoberta da Fundao Wikimedia utiliza um registro de eventos (Event Logging, EL) para avaliar vrias mtricas de performance e usabilidade, ajudando-os a tomar decises. Principalmente, este departamento est interessado em:

  • clickthrough rate: a proporo de sesses de busca onde o usurio clica em um dos resultados apresentados.

  • zero results rate: a proporo de buscas que no tiveram resultados.

e outras mtricas fora do escopo desta tarefa. EL usa Javascript para enviar mensagens aos nossos servidores de forma assncrona quando o usurio realiza aes especficas. Nesta tarefa voc vai analisar um subconjunto de nosso registro de eventos.

Dados

O conjunto de dados vem de uma estratgia de EL usada para avaliar a satisfao do usurio. Usurios standalone so randomicamente coletados para ser anonimamente rastreados por tal estratgia, que usa um sistema de descoberta do tipo Im alive para estimar quanto tempo os usurios permanecem nas pginas que eles visitam. O conjunto de dados contm em tabela pouco mais de uma semana de EL com as seguintes colunas.

Campo Tipo Descrio
uuid string Identificador universal nico (UUID) para gerenciamento dos registros.
timestamp integer Data e hora (UTC) do evento com a seguinte formatao: YYYYMMDDhhmmss.
session_id string Um identificador individual de sesso nico.
group string Uma marca (a ou b).
action string Identifica a natureza do evento em que foi criado. Veja abaixo.
checkin integer Quantos segundos a pgina passou aberta.
page_id string Um identificador nico relacionado s pginas visitadas e verificadas.
n_results integer Nmero de sugestes de pginas retornado. Apenas para eventos searchResultPage.
result_position integer A posio do link na SERP (Search Engine Results Page) para a pgina visitada.

O campo action pode ter os seguintes valores:

  • searchResultPage: quando uma nova busca realizada e uma SERP apresentado ao usurio.

  • visitPage: quando o usurio clica em um link dos resultados.

  • checkin: quando o usurio permaneceu em uma pgina por um intervalo de tempo pr-definido.

Questes

Q1. Qual a clickthrough rate geral diria? Como isto varia entre os grupos?

Algo crucial para comearmos a responder esta questo calcular a taxa de cliques por sesso de busca. Como a taxa por sesso, agrupar os dados desta forma um bom primeiro passo. Enquanto ocorre este agrupamento criamos duas novas variveis que ajudaram na tarefa de analisar a taxa de cliques: visited para identificar facilmente qual sesso teve ou no cliques (1 ou 0) e first_search_zero que marca aquelas sesses cuja primeira busca no trouxe resultados. results e num_clicks soma as variveis originais de mesmo nome, agrupando por sesso.

sessions = buscas %>% #agrupando por sesso
    group_by(group, day, session_id) %>% 
    summarise(visited = ifelse(sum(num_clicks) > 0, 1, 0),
              session_legth = first(session_legth),
              last_search = first(last_search),
              results = sum(results, na.rm = TRUE),
              num_clicks = sum(num_clicks, na.rm = TRUE), 
              first_search_zero = ifelse(first(results)==0, "Sim", "No"))

Partindo para a prtica usaremos um Stepped Line, que um tipo de grfico que reuni caractersticas de grfico de linha e histograma. Trata-se de um grfico de linha discreto, onde o eixo horizontal representa o tempo (em dias) e o eixo vertical exibe uma proporo calculada dividindo o total de sesses com buscas teis (com acesso a alguns dos links retornados) pelo total de sesses.

sessions %>% 
    group_by(group, day) %>% 
    summarise(proportion = sum(visited == 1)/n()) %>% 
        ggplot(aes(x = day, y = proportion, color = group)) + 
        geom_step() + 
    labs(x='Dias',  
        y="'clickthrough rate'",

    theme(plot.title = element_text(face="bold",size = "17"),

Sumrio

A partir das quatro questes propostas levantamos hipteses interessantes a respeito da natureza dos dados, chegamos a algumas concluses teis e pudemos entender melhor como estes esto correlacionados. Com base em Q1 e o clculo das sesses de buscas com acesso a resultados (cliques) pudemos ver uma grande discrepncia nas propores entre os grupos a e b, o que sugere uma diferena funcional entre estas duas categorias. Encontramos resultado atpicos como buscas sem resultados e, mesmo assim, com acesso a links (que no existiam). Alm de notarmos uma sutil melhora na taxa de cliques no meio da semana em relao ao fim de semana para a e o oposto para b.

Em Q2, percebemos que as pessoas tendem a dar preferncia s primeiras opes dos resultados de busca, como era de se esperar. Alm disto, observamos que, nos trs primeiros dias da amostra, h um volume muito maior de usurios que clicaram mais de uma vez nos resultados.
